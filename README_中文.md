# AITF — Agent Interaction Text Format（Agent交互文本格式）

**人类语言在agent间交互效率太低。这是一种提高agent间文本交互效率的标准化语言格式协议。**

---

## AITF 是什么？

AITF 是一套结构化语言规范，使 AI agent 能够以一致、可追溯、可机器解析的格式读写文档。它定义了一套共享的节点类型词汇表、状态码和结构规则——仅此而已。

任何 agent 加载此协议后，就能阅读任何 AITF 格式文档，也能写出任何 AITF 格式文档，不受文档用途限制。

---

## 解决什么问题？

当 AI agent 生成分析、从书中提取知识、向其他 agent 委派任务或提出解决方案时，输出通常是非结构化的自然语言。这带来几个问题：

**对机器而言：** 非结构化文本无法被可靠地解析、引用或在其基础上继续构建。当 Agent B 阅读 Agent A 的分析时，它没有系统化的方式来识别哪些是假设、哪些是证据、哪些是结论，哪些已被验证、哪些只是推测，以及推理链中的空白在哪里。

**对人类而言：** 审查 AI 生成的分析时，没有标准方式来查看推理链、质疑特定假设或追踪哪些内容已被独立验证。

**对协作而言：** 多个 agent 处理同一个问题时，如果没有共享格式，它们的贡献无法被清晰地分离、归属或协调。

## 设计原则

### 1. 一份协议，而非多份

此协议的早期版本按场景拆分规则——方案评审一套、知识提取一套、任务委派一套。这意味着 agent 必须先预判文档类型才能读取，而且不同文档类型使用不兼容的词汇表。

驱动当前设计的核心洞察：**阅读和写作不是分离的活动，场景是无限的。** Agent 读一份文档，形成自己的看法，然后写出一份包含原始内容和自己贡献的新文档。它读和写所遵循的规则是同一套规则——即表达语言本身。

因此 AITF 将一切统一为一份协议：一个节点类型注册表、一套状态码、一套结构规则。模板（常见场景的可选指引）在此之上，但协议本身与场景无关。

### 2. 节点是词汇，不是操作

协议定义的是**事物是什么**（标记为 `A` 的节点是假设，标记为 `K` 的节点是风险），而非**对它们做什么**。没有"质疑"或"修改状态"之类的"操作"——因为所有操作都归结为同一件事：写新的署名节点。

如果你不同意某人的主张，你写自己的节点表达不同意见。如果你验证了一个假设，你写一个节点说明验证依据。协议不需要枚举操作，因为词汇本身就足以表达任何行为。

### 3. 自我验证是不诚实的

一条核心规则：**作者永远不能给自己的假设、主张、解决方案、机制或风险标记为"已验证"。** 状态 `V`（已验证）意味着由另一方独立确认。自我验证等于没有验证。

这条规则不是建议——它内嵌在状态分配规则中。它强制要求智识上的诚实：每份文档都公开展示哪些已被独立检验、哪些没有。

### 4. 文档自包含

每份 AITF 文档必须让任何阅读它的 agent 无需外部信息即可完全理解。不依赖聊天记录、消息队列或"你当时在场才懂"。一个文件 = 一个完整上下文。

### 5. 只追加、必署名

任何 agent 不修改另一个 agent 的内容。如有异议，写自己署名的节点。这创造了不可篡改的审计链：你总能追溯谁在什么时间写了什么、为什么写。版本历史记录在强制要求的变更日志中。

## 节点类型注册表

协议提供四类节点类型。Agent 使用最能表达内容的任何类型——类型**不限定于特定场景**。

### 分析类节点
用于推理、论证和解决方案。

| 代码 | 含义 |
|------|------|
| P | 问题陈述 |
| E | 证据（数据、引用、观察） |
| A | 假设（未经直接证明但被当作成立的前提） |
| C | 主张（从证据和假设推导的结论） |
| S | 解决方案（提议的行动） |
| M | 机制（解决方案如何运作） |
| K | 风险（可能出错的地方） |
| R | 规则（约束或原则） |
| D | 依赖（前置条件） |
| G | 空白（缺失的信息） |

### 知识类节点
用于从书籍、论文或文档中提取的结构化知识。

| 代码 | 含义 |
|------|------|
| CON | 概念（独立知识单元） |
| DEF | 定义 |
| REL | 关系（概念之间） |
| PRO | 属性（概念的特征） |
| EXA | 示例 |
| CTR | 反例（边界条件或例外） |
| APP | 应用（有用的场景） |
| SRC | 来源引用 |
| TAG | 检索标签（搜索优化关键词） |
| GAP | 知识空白 |

### 任务类节点
用于工作委派和执行追踪。

| 代码 | 含义 |
|------|------|
| OBJ | 目标（任务目标） |
| CTX | 背景（为什么需要这个任务） |
| IN | 输入（文件、数据、先前结果） |
| OUT | 输出（预期交付物） |
| CST | 约束（不能做什么） |
| ACC | 验收标准 |
| DEP | 依赖项（前置条件） |
| Q | 问题（请求澄清） |
| LOG | 执行日志 |

### 协作类节点
用于 agent 间对话。

| 代码 | 含义 |
|------|------|
| SUG | 建议 |
| RSP | 回应 |
| JUS | 理由（决策的推理依据） |
| MOD | 修改版（建议的改编版本） |

---

## 状态码

每个节点都携带一个状态，诚实地表示其验证状态。

| 代码 | 含义 |
|------|------|
| V | 已验证——已被独立确认 |
| U | 未检验——已提出但尚未审查 |
| X | 有争议——存在冲突的证据 |
| N | 无证据——占位或推测性内容 |
| F | 人工标记——需要人类判断 |
| D | 已完成——已完成 |
| B | 被阻塞——等待依赖项 |
| W | 已撤回——被作者撤回 |

## 质量标准

协议内置了质量保障机制：

**自检要求：** 每份文档在定稿前必须通过属性完整性、引用完整性和状态规则合规性检查。

**六种反模式**，作者必须主动避免：
1. 信心膨胀（高信心但缺乏强力证据）
2. 假设隐藏（假设埋在主张内部）
3. 风险盲视（解决方案没有风险分析）
4. 空白回避（复杂文档中没有承认的未知项）
5. 循环论证（主张基于的假设来源于该主张本身）
6. 过早方案化（在界定问题之前就提出解决方案）

**诚实自评：** 对实质性文档，作者需识别出最薄弱的节点、最不确定的假设和最大的空白。

## 模板

模板是常见场景的可选效率指引。它们建议典型的节点组合和执行流程，但不增加新规则——所有描述都使用同一套核心协议词汇。

| 模板 | 典型用途 |
|------|---------|
| [方案生成](template_proposal_authoring.md) | Agent 从自身推理出发生成结构化方案 |
| [知识提取](template_knowledge_extraction.md) | 将书籍或文档转化为结构化知识 |
| [任务委派](template_task_delegation.md) | 在 agent 之间委派和追踪工作 |
| [方案评审](template_proposal_review.md) | 评估和批判已有方案 |
| [建议](template_suggestion.md) | agent 之间的轻量级改进提议 |

## 为什么不直接用自然语言？

自然语言富有表达力但充满歧义。当一个 agent 写"这个方案应该效果不错"时，你无法系统性地判断：这是经过验证的主张还是假设？有什么证据支持它？agent 考虑过哪些风险？有人独立验证过吗？

AITF 使这些区分变得显式且可机器解析，同时保持人类可读。结构化格式不是推理的替代品——它是让推理变得透明的方式。

## 架构演进

此协议经历了四个主要版本的演进，每一次都由实际问题驱动：

**v2.0** ——按场景拆分（方案、任务、知识、建议各有独立扩展）。每个扩展定义自己的节点类型和操作。

**v3.0** ——填补 v2.0 的空白（缺失的操作格式、不完整的执行流程、薄弱的终止条件）。新增方案生成扩展。

**v3.0 → v4.0 突破** ——一系列第一性原理的质疑揭示了按场景拆分是错误的抽象层：

- *"如果一个 agent 读了一份方案然后又读了一本书，为什么需要两套不同的协议？"* ——节点词汇不应该锁定在特定场景里。
- *"生成方案、提取知识、提建议、派任务——这些都只是在写。"* ——这些'操作'之间没有本质区别。
- *"读和写遵循相同的规则。"* ——能让你理解一份文档的词汇，也是让你写出一份文档的词汇。
- *"增加、删除、修改和新建本质上都是'新写'。"* ——只追加规则意味着所有操作都是同一个原语。

**v4.0** ——将一切统一为一套表达语言（约 3,400 tokens）。模板变为可选附加项（每个约 600–800 tokens），不再是协议组件。

## 作者

设计者：**卓明海（Minghai Zhuo）**

AITF 汲取了智能主义（Intelligenism）的原理——一个研究智能如何通过自下而上的协作在不同尺度上涌现的理论框架，从神经元到大脑、从个体到组织、从 agent 到 agent 网络。

协议应该定义*如何表达*，而非*做什么*——这一核心洞察，与智能主义的原则一脉相承：组织智能源于共享的语言和结构，而非自上而下的命令。
