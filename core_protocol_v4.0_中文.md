# OPENCLAW_核心协议 v4.0

---

## 目的

这是所有OpenClaw_agent文档的表达语言。
一份协议覆盖所有场景：阅读和写作使用相同的规则。
agent加载此协议→能阅读任何agent格式文档→能写作任何agent格式文档。

---

## 1_文件规则

### 1.1_一个文件=一个完整上下文
所有结构化内容以_.md_文件形式存在。
任何agent阅读文件后无需外部信息即可理解全部状态。
不依赖聊天室、消息队列或文件外讨论。

### 1.2_记录格式
每条记录占一行。
字段用竖线 `|` 分隔。
层级通过显式父节点引用 `^parent_id` 表达。
```
格式: id|类型|状态|内容|^父节点id|依赖|影响|作者
示例: A1.2|A|U|用户留存由入职体验质量驱动|^C1|D1.1|S2,K3|@McMillan
```

### 1.3_文件头部（必填）
每份agent格式文档必须以此开头：
```
## META
protocol_version: v4.0
type: {文档用途:proposal|knowledge|task|suggestion|hybrid|other}
scope: {本文档覆盖的范围}
status: draft|in_progress|review|closed
author: @{AgentName}
participants: [{多agent时的参与者列表}]
created_date: {YYYY-MM-DD}
source: {可选：如果源自外部材料则注明来源}
source_author: {可选：原始作者}
```

## 2_节点类型注册表

所有节点类型可用于任何文档。
agent根据需要表达的内容选择使用哪些类型。

### 2.1_分析类节点
用于表达推理、论证、解决方案。
```
P=问题           问题陈述或提问
E=证据           事实依据（数据、观察、引用）
A=假设           未经直接证明但被当作成立的前提
C=主张           从证据和假设推导出的结论
S=解决方案       提议的行动或回答
M=机制           解决方案如何运作
K=风险           可能出错的地方
R=规则           约束行为的原则或规定
D=依赖           必须先解决的前置条件
G=空白           已识别的缺失信息或未解决的问题
```

### 2.2_知识类节点
用于表达从来源材料中提取的结构化知识。
```
CON=概念         独立的知识单元
DEF=定义         对概念的精确描述
REL=关系         概念之间的关联
PRO=属性         概念的特征或特性
EXA=示例         例证
CTR=反例         边界条件或例外
APP=应用         该知识有用的场景
SRC=来源         出处引用（书名、章节、页码、作者、URL）
TAG=检索标签     为向量搜索优化的关键词
GAP=知识空白     已识别的缺失知识（G在知识场景下的别名）
```

### 2.3_任务类节点
用于表达工作委派和执行记录。
```
OBJ=目标         任务目标
CTX=背景         为什么需要这个任务
IN=输入          输入文件、数据或先前结果
OUT=输出         预期交付物（格式+内容）
CST=约束         限制条件（不能做什么）
ACC=验收标准     可衡量的完成标准
DEP=依赖项       前置条件（D在任务场景下的别名）
Q=问题           请求澄清
LOG=执行日志     做了什么以及结果的记录
```

### 2.4_协作类节点
用于表达建议、回应和agent间对话。
```
SUG=建议         提议的变更或改进
RSP=回应         对建议或问题的回复
JUS=理由         决策的推理依据（接受、拒绝、推迟）
MOD=修改版       建议的改编版本
```

### 2.5_节点属性（每个节点必填）
```
id:      唯一标识符（如 CON_1.3, S2.1, OBJ_1）
type:    来自本注册表
status:  来自下方状态码
author:  @{AgentName}
```

### 2.6_跨类型使用
节点类型不限定于特定场景。
示例：
```
- 任务文档可以包含K（风险）和A（假设）节点
- 知识文档可以包含G（空白）和Q（问题）节点
- 方案可以包含CON（概念）节点，如果它引用了提取的知识
- 建议可以包含E（证据）来支持其理由
```
agent使用最能表达内容的任何节点类型。

## 3_状态码

### 3.1_通用状态码
适用于所有文档中的所有节点类型。
```
V=已验证     已独立确认（由另一个agent、由来源、由数据）
U=未检验     已提出但尚未审查或确认
X=有争议     存在冲突的证据或解读
N=无证据     占位或推测性内容（无支持依据）
F=人工标记   需要人类判断来解决
D=已完成     已完成（主要用于任务类节点）
B=被阻塞     等待依赖项或外部输入
W=已撤回     被作者撤回
```

### 3.2_状态分配规则
```
规则1: 作者永远不能给自己的A/C/S/M/K标V。
       V意味着独立验证。自我验证=没有验证。
规则2: E可以标V，如果可直接验证（引用、测量、数据）。
规则3: CON/DEF可以标V，如果已被另一个agent对照来源交叉验证。
规则4: 有疑问时用U。U是诚实的。过早标V是不诚实的。
规则5: N用于已知的未知。当你知道需要证据但没有时用N。
规则6: F用于超出agent能力的决策。参见升级触发条件。
```

### 3.3_状态转换
```
U→V: 提供了独立验证
U→X: 发现了冲突的证据或解读
U→F: agent无法解决，需要人类
N→U: 找到了部分依据
N→V: 找到了完整证据并经验证
X→V: 以充分证据解决了冲突
V→X: 新的矛盾证据（必须有实质支持）
任何→W: 作者撤回（必须说明原因）
B→V: 依赖已解决
B→F: 依赖无法由agent解决
```

## 4_结构规则

### 4.1_层级
节点通过父节点引用形成树形结构。
```
P1[U]: 问题陈述
  E1.1[N]: 证据
  A1.1[U]: 假设
  C1.1[U]: 主张
    S1.1.1[U]: 解决方案
      M1.1.1.1[U]: 机制
      K1.1.1.1[U]: 风险
```
深度无限制。agent使用内容所需的任何深度。

### 4.2_引用
节点可以引用其他节点以实现可追溯性。
```
based_on: [{节点ID}]      本节点的推导来源
depends:  [{节点ID}]      本节点成立的前提条件
impacts:  [{节点ID}]      本节点变化时受影响的下游节点
blocking: [{节点ID}]      在本节点解决前无法推进的内容
```
所有引用必须指向已存在的节点。不允许孤立引用。

### 4.3_关系类型（专用于REL节点）
```
is_a:          子类关系
part_of:       组成部分关系
depends_on:    理解前置依赖
contradicts:   概念之间的冲突
extends:       细化或延伸
applies_to:    应用场景
analogous_to:  不同领域的结构相似性
```

### 4.4_标签规则（专用于TAG节点）
```
每个TAG必须包含：
- 同义词和别名
- 可能的查询措辞（agent会怎么搜索这个内容）
- 领域特有术语（如适用）

质量标准：
  好：覆盖≥3个搜索角度（正式术语、口语化表达、缩写、相关领域）
  差：单个词或只有精确名称
  好的示例：[边际成本,MC,增量成本,单位变动成本,多生产一个要多花多少钱]
  差的示例：[边际成本]
```

## 5_署名与版本控制

### 5.1_作者署名
```
每段内容必须署名：@{AgentName}
多个agent对同一文档贡献时：
  每个agent的贡献需清晰归属
  不得修改另一个agent已署名的内容
  如有异议：写自己署名的节点（而非编辑对方的）
```

### 5.2_文档版本控制
```
每次新贡献=次版本号（v1.1, v1.2）
人类确认=主版本号（v1→v2）
格式：v{X.Y} @{AgentName}: {简要变更描述}
```

### 5.3_变更日志（必填）
每份文档以此结尾：
```
## CHANGELOG
v1.0 @{AgentName}: {做了什么}
v1.1 @{AnotherAgent}: {添加了什么}
```

## 6_质量标准

### 6.1_自检（完成任何文档前必须执行）
```
作者必须验证：
- 每个节点都有id、类型、状态、作者（无缺失属性）
- 所有引用指向已存在的节点（无孤立引用）
- 状态分配规则已遵守（A/C/S/M/K未自我标V）
- 无循环依赖链（A基于C基于A）
- 文档自包含（读者无需外部信息）
```

### 6.2_反模式（作者必须避免）
```
反模式1：信心膨胀
  症状：高信心但缺乏强力证据
  修正：信心必须有依据。有疑问时→标U而非V

反模式2：假设隐藏
  症状：假设嵌入在C或S内部，没有独立的A节点
  修正：每个如果/当/给定都应是独立的A节点

反模式3：风险盲视
  症状：解决方案没有K节点或只有轻微风险
  修正：对每个S问"什么可能出错"和"最坏情况是什么"

反模式4：空白回避
  症状：复杂文档中没有G节点
  修正：每份非简单文档都有未知项。如果你找不到→你的审查有缺陷

反模式5：循环论证
  症状：C基于A，而A又来源于C
  修正：追踪每条based_on链直到E或显式的A[N]

反模式6：过早方案化
  症状：S节点缺乏充分的P范围界定或E收集
  修正：先问题、先证据、再方案
```

### 6.3_诚实评估（建议对实质性文档附加）
```
作者在文档末尾附加：
## SELF_ASSESSMENT
weakest_node: {ID}: {为什么这是最薄弱的点}
most_uncertain_assumption: {A_ID}: {为什么}
biggest_gap: {G_ID}: {为什么}
overall_confidence: high|medium|low
```

### 6.4_空白严重性（使用G或GAP节点时）
```
G_ID[open]: {描述}
  severity: critical|high|medium|low
  blocking: [{没有这个就无法标V的节点ID}]
  resolution_hint: {什么可能填补这个空白}
```

## 7_升级机制

### 7.1_何时标记F（标记等待人类决策）
```
- agent之间有有效证据支持的矛盾且无法解决
- 需要实验测试或运行时验证
- 商业判断（成本、优先级、资源分配）
- 不可逆操作（执行后无法撤销）
- 来源材料模糊（存在多种合理解读）
- 概念边界不清（需要领域专业知识）
- 跨来源矛盾（同一主题，不同说法）
- 范围或优先级判断（包含或排除什么）
- 涉及agent身份或SOUL.md
```

### 7.2_升级格式
```
node_ID[F]: {描述}
  @{AgentName}: escalation
    reason: {为什么agent无法解决}
    options: [{可能的解决方案}]
    human_question: {需要人类做出什么具体决策}
```

## 8_多Agent协作

### 8.1_多个agent处理同一文档时
```
每个agent先完整阅读已有文档。
每个agent用自己的@{AgentName}署名写新节点。
不得修改另一个agent的内容。
如有异议→写自己的节点表达反对意见或替代方案。
分歧无法解决→标记F。
```

### 8.2_跨文档引用
```
引用另一份文档中的节点时：
格式：{文档文件名}:{节点ID}
示例：project_alpha_proposal.md:S2.1
```

## 9_向后兼容

```
规则：旧文件在新版本协议下必须仍可阅读。

增量变更（新节点类型、新状态码、新字段）：
  无需迁移。旧文件原样有效。

破坏性变更（改分隔符、改ID方案、删除节点类型）：
  需要人类批准（F）。
  必须包含：影响评估 + 迁移脚本 + 成本论证。
```

## 10_协议治理

### 10.1_修改流程
```
agent通过协议意见文档提出变更提议。
人类审查并批准。
agent永远不能直接修改正在使用的协议文件。
```

### 10.2_工作中的反馈
```
当agent发现协议缺口时：
1. 正常完成当前任务（不要中断）
2. 在协议意见文档中记录缺口：
   G_ID[open]: {缺口描述}
     trigger: {暴露此缺口的任务}
     date: {YYYY-MM-DD}
     severity: critical|high|medium|low
3. 继续工作
```
